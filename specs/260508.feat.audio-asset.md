# 支持音频素材

## 背景

- 当前已经实现基础的时间轴交互、预览与导出
  - demo @src/demo/App.tsx
  - timeline @src/lib/Timeline.tsx
  - 预览播放 @src/lib/PreviewPlayer.tsx
  - 导出 @src/lib/video-exporter.ts
  - 数据类型 @src/lib/types.ts

## 需求

- 支持在时间轴添加音频素材
  - 复用已有实现，绘制波形图 @src/lib/WaveformCanvas.tsx
- 预览播放支持在 start 时间加载并播放音频素材 @src/lib/PreviewPlayer.tsx
- 音频素材的播放，也需支持 seek 指定时间定启播
- 导出的视频需混入音频素材，截取掉音频素材与 deletedRanges 重叠部分
  - 截取方式参考视频音轨的实现

## 现状分析

### 类型系统

- `types.ts` 已定义 `IAudioItem`（`type: "audio"`, 继承 `IMediaItem` 含 `file` 字段）
- `Item` 联合类型已包含 `IAudioItem`，类型层面已就绪
- `IMainTrackConf.item` 支持 `IVideoItem | IAudioItem`

### 时间轴渲染

- `Timeline.tsx` 仅为 mainTrackConf 的主轨创建 `WaveformExtractor`
- `TimelineRow.tsx` 对主轨音频已使用 `AudioTrackItem` 渲染波形
- **问题**: overlay 层（`layers` 数组中的 item）统一使用 `TrackItem` 渲染，无论类型，音频 overlay 项只显示纯色块，无波形

### 预览播放

- `PreviewPlayer.tsx` 仅处理 `mainTrackConf.item`（主轨），不感知 overlay items
- 已支持 video/audio 两种主轨类型，具备 seek、play/pause、跳过 deletedRanges 能力
- **问题**: 不支持播放 overlay 层中的音频素材

### 视频导出

- `video-exporter.ts` 仅从主轨文件导出视频帧和音频采样
- 已有 `computeValidSegments` 处理 deletedRanges，已有 `computeTimeOffset` 做时间重映射
- **问题**: 不处理 overlay 层音频素材的混入

### 波形提取

- `WaveformExtractor` 支持任意 `File` 输入，`extract(start, end, pps)` 接口通用
- 当前仅对主轨创建一个实例，overlay 音频项无 extractor

### Demo 应用

- `App.tsx` 的 `handleImportItems` 已支持导入音频文件作为 overlay items
- 但导入后在时间轴上无波形显示，预览和导出也不处理

### Bug1: overlay 音频素材不可拖拽

- `AudioTrackItem.tsx` 不接受 `onDragStart` prop，也没有 `onMouseDown` 拖拽处理
- `TrackItem.tsx` 有 `onDragStart` + `handleMouseDown` 支持拖拽
- `TimelineRow.tsx:266-285` 对 overlay 音频渲染 `AudioTrackItem` 时未传递 `onDragStart`
- 对比：当 overlay 音频尚无 waveformExtractor 时，走 `TrackItem` 分支反而可拖拽，行为不一致

### Bug2: 添加/删除 overlay 素材导致缩略图和波形全部刷新

- 根因：`App.tsx` 的 `handleImportItems` / `handleDeleteItem` 调用 `bumpVersion()`，将整个 `<Timeline>` 组件卸载再重建
- `bumpVersion` 实现（`App.tsx:120-123`）：`setShowTimeline(false)` → `queueMicrotask(() => setShowTimeline(true))`
- 卸载重建导致所有 `ThumbnailExtractor` / `WaveformExtractor` 销毁重建、所有 `<canvas>` 重绘
- `Timeline.tsx:33` 使用 `createSignal(props.initData.items)` 初始化 items，不响应 props 变化
- 因此 `App.tsx` 通过 `bumpVersion` 强制重建以同步 props 变更——这对 overlay items 的增删是过度行为

### Bug3: AudioTrackItem 波形提取使用时间线绝对时间，导致 overlay 音频波形错误

- **根因**：`AudioTrackItem.tsx:loadWaveform()` 将 `props.visibleStart` / `props.visibleEnd`（时间线绝对时间）直接传给 `WaveformExtractor.extract()`，但后者 `rawPeaks` 是从文件开头(0s)索引的，期望接收文件相对时间
- **影响**：overlay 音频 item 的 `startTime ≠ 0` 时，会提取到错误的波形数据甚至超出文件长度返回空数据
- **主轨道不受影响**：主轨道 `mainTrackItem.startTime` 通常为 0，`visibleStart` 恰好等于文件偏移量
- **具体场景**：音频文件 5s，overlay item 置于 `startTime=3`，`extract(3, 8, pps)` 尝试提取文件 3~8s，但文件只有 0~5s，应提取 0~5s
- **次要问题**：`createEffect` 未追踪 `visibleStart`/`visibleEnd`，拖拽导致可见范围变化时波形数据不重新提取（跨行移动场景）→ 已由任务13修复

### Bug5: PreviewPlayer 播放 overlay 音频时机错误

- **现象**：预览播放一开始就播放 overlay 音频，而不是等 currentTime 到达素材 startTime 时再播放
- **根因**：App.tsx 的 `onChange` 回调仅同步 `deletedRanges`，未同步 `items`
  - Timeline 拖拽 overlay 音频后，内部 items 的 startTime 已更新并通过 `emitChange`（Timeline.tsx:242）传出
  - 但 App.tsx onChange（App.tsx:408-411）只执行 `setDeletedRanges(arg.deletedRanges ?? [])`，忽略了 `arg.items`
  - PreviewPlayer 接收 `items={data()!.items}`（App.tsx:434），始终使用导入时的原始 startTime（通常为 0）
  - 导致 `syncOverlayAudio` 中 `inRange = mainTime >= item.startTime` 在播放起始（mainTime=0）时就为 true，立即播放音频
- **主轨道不受影响**：主轨道时间由 HTMLMediaElement 的 currentTime 直接驱动，不依赖 items 数据

### Bug6: 拖拽音频素材时波形图不必要刷新

- **现象**：拖拽 overlay 音频素材改变 startTime 时，波形图被重新生成（视觉闪烁）
- **根因**：AudioTrackItem.tsx 的 createEffect 追踪 `visibleStart`/`visibleEnd`（时间线绝对时间）
  - 拖拽时绝对时间变化触发 `throttledLoad()`（300ms 延迟后提取波形）
  - 但文件相对时间（`visibleStart - item.startTime`）实际未变——拖拽使两者等量偏移
  - `loadWaveform()` 重新 `extract()` 并 `setWaveformData(new Float32Array)`，新引用触发 SolidJS 重渲染
  - WaveformCanvas 组件因 waveformData 引用变化而重绘 canvas，造成视觉闪烁
- **应跳过刷新的场景**：item 在单行内水平拖拽，文件相对可见范围未变化
- **仍需刷新的场景**：item 跨行移动导致可见范围变化，此时文件相对时间确实改变

### Bug4: AudioTrackItem 未接收 item prop，无法获取 startTime 做时间偏移

- `TimelineRow.tsx:276-284` 对 overlay 音频渲染 `AudioTrackItem` 时，当前仅传 `extractor`/`visibleStart`/`visibleEnd`/`left`/`width`/`pixelsPerSecond`
- 未传 `item` prop，`AudioTrackItem` 内部无法获取 `item.startTime` 来计算文件相对时间
- 主轨道 AudioTrackItem（`TimelineRow.tsx:216-225`）同样未传 `item`，但因为主轨 startTime 通常为 0 所以无影响

## 技术实现方案

### 方案一：Timeline - overlay 音频项渲染波形

**核心变更**: 在 `Timeline.tsx` 中为每个 overlay 音频项创建并管理 `WaveformExtractor` 实例，传递给 `TimelineRow` 用于渲染。

1. `Timeline.tsx` 新增 `waveformExtractors` signal（`Map<string, WaveformExtractor>`），按 item id 管理
2. 在 `items` 变化时，为新增的 `type === "audio"` 的 overlay item 创建 `WaveformExtractor`，移除已删除项的 extractor
3. `TimelineRow` props 新增 `itemWaveformExtractors: Map<string, WaveformExtractor>`
4. `TimelineRow.tsx` overlay 渲染逻辑：当 `slice.item.type === "audio"` 且存在对应 extractor 时，渲染 `AudioTrackItem` 而非 `TrackItem`

### 方案二：PreviewPlayer - 支持多音频播放

**核心变更**: `PreviewPlayer` 接收 `items` 列表，为每个音频 item 创建 `HTMLAudioElement`，在对应时间点播放。

1. `PreviewPlayerProps` 新增 `items: Item[]` 属性
2. 内部维护 `Map<string, HTMLAudioElement>` 管理 overlay 音频元素
3. 播放逻辑:
   - 主轨播放时，RAF tick 中检查每个音频 item 是否应在当前时间播放
   - 若 `currentTime >= item.startTime && currentTime < item.endTime`，确保对应 audio 元素正在播放，并同步 `audio.currentTime = currentTime - item.startTime`
   - 若不在范围内，暂停对应 audio
4. seek 逻辑:
   - 主轨 seek 时，同步 seek 所有音频 item 的 audio 元素
   - `audio.currentTime = Math.max(0, seekTime - item.startTime)`
   - 若 seekTime 在音频范围内则播放，否则暂停
5. deletedRanges 处理:
   - 音频 item 落入 deletedRange 的部分静音，参考主轨 `skipIfDeleted` 逻辑
   - 简化方案：音频 item 的播放位置如果处于 deletedRange 内，跳到 range.end

### 方案三：Video Exporter - 混入音频素材

**核心变更**: `exportVideo` 函数接收 overlay 音频 items，将其音频采样混入输出。

1. `exportVideo` 签名新增 `audioItems?: IAudioItem[]` 参数
2. 导出流程:
   - 先处理主轨视频帧（同现有逻辑）
   - 再处理主轨音频采样（同现有逻辑）
   - 最后处理每个 overlay 音频 item:
     - 为每个音频 item 创建独立的 `Input` + `AudioSampleSink`
     - 遍历 `computeValidSegments` 计算有效时间段
     - 对每个有效 segment，从音频 item 中提取对应时间段的采样
     - 将采样时间重映射到输出时间轴（加上 `item.startTime` 偏移，再减去 deletedRanges 偏移）
     - 混入 `pipeline.audioSource`
3. 时间映射:
   - 音频 item 在时间轴上的绝对时间 `t` 对应音频文件内的时间 `t - item.startTime`
   - 需要 `item.startTime + itemOffset` 映射到输出时间轴的 `t - computeTimeOffset(t, deletedRanges)`

### 方案四（Bug1）: AudioTrackItem 支持拖拽

**核心变更**: 为 `AudioTrackItem` 增加 `onDragStart` 回调，与 `TrackItem` 行为一致。仅支持位置拖拽（改变 startTime），不支持 Trim/Resize。

1. `AudioTrackItemProps` 新增 `onDragStart?: (e: MouseEvent, item: Item) => void` 和 `item: Item`
2. `AudioTrackItem` 模板根元素添加 `onMouseDown` 处理函数，调用 `props.onDragStart`
3. `AudioTrackItem` 根元素添加 `cursor: grab` 样式（有 onDragStart 时）
4. `TimelineRow.tsx` 传递 `onDragStart={props.onItemDragStart}` 和 `item={slice.item}` 给 `AudioTrackItem`

### 方案五（Bug2）: Timeline 响应式同步 props，消除 bumpVersion

**核心变更**: `Timeline.tsx` 对 `props.initData.items` 变化做增量同步，`App.tsx` 对 overlay items 增删不再调用 `bumpVersion`。同一 id 冲突时内部保留（不覆盖用户拖拽位置）。

1. `Timeline.tsx` 新增 `createEffect(on(() => props.initData.items, ...))`：
   - 比较 props.initData.items 与当前内部 items 的 id 差异
   - 仅增量更新：追加新增项、移除已删项，已有项保持不变（保留用户拖拽位置）
   - `setItems` 触发后，overlay waveform extractor effect 自然响应新 items
2. 同理对 `props.initData.deletedRanges` 做响应式同步（可选，当前 deletedRanges 仅由 Timeline 内部管理）
3. `App.tsx`:
   - `handleImportItems` 不再调用 `bumpVersion()`
   - `handleDeleteItem` 不再调用 `bumpVersion()`
   - `handleReplaceMainTrack` / `handleImportAsr` 等涉及 mainTrackConf 变更的操作保留 `bumpVersion`（extractor 需重建）
4. `App.tsx` 中 `Timeline` 组件从 `Show when={showTimeline()}` 条件渲染改为始终渲染（可选，取决于是否保留 bumpVersion 用途）

### 方案六（Bug3/4）: AudioTrackItem 波形提取时间偏移修正

**核心变更**: `AudioTrackItem.loadWaveform()` 将时间线绝对时间转为文件相对时间后再传给 `extractor.extract()`；`createEffect` 追踪 `visibleStart`/`visibleEnd` 变化重新加载波形。

1. `AudioTrackItemProps` 确认已有 `item?: Item` prop（任务8 已添加）
2. `AudioTrackItem.tsx` `loadWaveform()` 修改：
   - 计算 `itemStartTime = props.item?.startTime ?? 0`
   - 传入 `extract(itemStartTime > 0 ? props.visibleStart - itemStartTime : props.visibleStart, itemStartTime > 0 ? props.visibleEnd - itemStartTime : props.visibleEnd, pps)`
   - 即将时间线绝对时间转换为文件相对时间
3. `AudioTrackItem.tsx` `createEffect` 追踪列表新增 `() => props.visibleStart` 和 `() => props.visibleEnd`：
   - 当 `visibleStart`/`visibleEnd` 变化时，重新调用 `throttledLoad()` 提取波形数据
4. `TimelineRow.tsx` 确认 overlay AudioTrackItem 已传 `item={slice.item}`（任务9 已完成）

### 方案七（Bug5）: App.tsx onChange 同步 items

**核心变更**: App.tsx 的 onChange 回调同时同步 items 到 data() signal，使 PreviewPlayer 获得拖拽后的正确 startTime。

1. App.tsx onChange 回调中，当 `arg.items` 存在时更新 `data()` 的 items：
   - `setData(d => d ? { ...d, items: arg.items! } : d)`
2. Timeline 内部拖拽已通过 `emitChange`（Timeline.tsx:59-68）传出 items，无需修改
3. PreviewPlayer 已通过 `items={data()!.items}`（App.tsx:434）响应式绑定，data 更新后自动获取最新 startTime
4. Timeline 的 `createEffect(on(props.initData.items, ...))` 会因 props 变化触发，但比较 id 后无增删，直接 return，不会覆盖内部拖拽状态

### 方案八（Bug6）: AudioTrackItem 使用文件相对时间判断是否需要重新提取

**核心变更**: 将 `lastDrawnVisibleStart`/`lastDrawnVisibleEnd`（绝对时间）替换为 `lastDrawnFileStart`/`lastDrawnFileEnd`（文件相对时间），避免单行内拖拽时不必要的波形重新提取。

1. `AudioTrackItem.tsx` 中将 `lastDrawnVisibleStart`/`lastDrawnVisibleEnd` 变量改为 `lastDrawnFileStart`/`lastDrawnFileEnd`
2. `loadWaveform()` 中计算 `fileStart`/`fileEnd` 后更新 `lastDrawnFileStart`/`lastDrawnFileEnd`
3. createEffect 中计算当前 `fileStart = props.visibleStart - (props.item?.startTime ?? 0)` 和 `fileEnd`，比较与上次是否变化
4. 仅当 `fileStart`/`fileEnd` 或 `pixelsPerSecond` 变化时才调用 `throttledLoad()`
