## 需求

- 引入 mediabuny 库，提取视频的缩略图，示例代码：

```ts
import { Input, ALL_FORMATS, BlobSource, CanvasSink } from "mediabunny";

const input = new Input({
  formats: ALL_FORMATS,
  source: new BlobSource(file), // videoItem.file
});

const videoTrack = await input.getPrimaryVideoTrack();
const sink = new CanvasSink(videoTrack, {
  width: 320, // 缩略图宽度
  height: 240, // 缩略图高度
  fit: "contain", // 缩放方式  fill' | 'contain' | 'cover'
});

// 生成多个等间距的缩略图
const startTimestamp = await videoTrack.getFirstTimestamp();
const endTimestamp = await videoTrack.computeDuration();
const timestamps = [0, 0.2, 0.4, 0.6, 0.8].map(
  (t) => startTimestamp + t * (endTimestamp - startTimestamp),
);

for await (const result of sink.canvasesAtTimestamps(timestamps)) {
  // 处理每个缩略图
  result.canvas; // 缩略图画布
}
```

- 时间轴上视频元素的高度设置为 40px，填充上述代码提取的缩略图
- 因为视频时间可能较长，帧数超级多、提取成本较大；你需要支持懒加载（加载可见区间的缩略图），以及每张缩略图的间隔时间（步长）
  - 第一步 f（Pixels per second， container width，video{ width， height, duration }） => { startTime， endTime， step }
  - 第二步 使用 mediabunny 提取视频缩略图 { startTime， endTime， step } => thumbnails
  - 第三步 将缩略图渲染到时间轴上
    1. 获取视频元素中时间轴上的多行总长度，创建一个 OffscreenCanvas
    2. 获取的缩略图渲染到 OffscreenCanvas，横向排列
    3. 将 OffscreenCanvas 内容渲染到时间轴中对应的视频元素节点上，注意视频元素可能跨行
  - 当 Pixels per second 变更时，重复提取逻辑，注意需要 throttle

## 现状分析

### 当前代码结构

- **TrackItem.tsx**（47行）：视频元素当前渲染为纯色矩形（`#4A3728`），高度固定 `ROW_ITEM_HEIGHT = 32px`
- **constants.ts**：`ROW_ITEM_HEIGHT = 32`，需要为视频主轨单独设置 40px
- **TimelineRow.tsx**：主轨渲染在 `mainTrackTop()` 位置，使用 `TrackItem` 组件，高度计算依赖 `ROW_ITEM_HEIGHT`
- **Timeline.tsx**：管理折行布局、拖拽、框选等交互；通过 `rows()` memo 计算折行；`pps()` 获取当前缩放级别
- **layout.ts**：`computeRows()` 计算折行，`assignItemsToRows()` 将 Item 分配到行（处理跨行拆分）
- **types.ts**：`IVideoItem` 继承 `IMediaItem`，有 `file: File` 字段，可直接传给 mediabunny 的 `BlobSource`

### 需要变更的关键点

1. **视频主轨高度**：从 32px 改为 40px，影响 `constants.ts`、`TimelineRow.tsx` 中的高度计算
2. **新增 mediabunny 依赖**：package.json 中目前无此依赖
3. **缩略图提取逻辑**：需新建模块，封装 mediabunny 调用 + 缓存
4. **缩略图渲染**：新建 VideoTrackItem 组件替代主轨的 TrackItem
5. **跨行处理**：视频 Item 被 `assignItemsToRows` 拆分为多个 `RowItemSlice`，每行的缩略图需要正确裁剪对应时间段

### 缩略图步长计算逻辑

给定 `pixelsPerSecond`（pps）和视频宽高比，缩略图高度固定 40px，则：

- 单张缩略图宽度 = `40 * (videoWidth / videoHeight)`（保持视频宽高比）
- step（时间步长）= `thumbnailWidth / pps`（每张缩略图覆盖的时间）
- 可见区间由当前折行的 `startTime` / `endTime` 决定（所有行的并集 = 整个视频时长）

### 渲染方案

采用 OffscreenCanvas 方案：

1. 计算视频在所有行中的总像素宽度 = `duration * pps`
2. 创建一个 OffscreenCanvas（宽 = 总像素宽度，高 = 40px）
3. 将提取到的缩略图按时间顺序横向绘制到 OffscreenCanvas 上
4. 每行的主轨 VideoTrackItem 通过 `drawImage` 裁剪 OffscreenCanvas 的对应区域进行渲染

由于折行布局所有行都可见（无滚动），只需加载整个视频时间范围的缩略图。当 pps 变化时 throttle 重新提取。

## 技术实现方案

### 1. 安装 mediabunny 依赖

在 package.json 中添加 `mediabunny` 为生产依赖（最新版）。

### 2. 视频主轨高度调整

- 在 `constants.ts` 新增 `MAIN_TRACK_HEIGHT = 40`
- `TimelineRow.tsx` 中主轨高度计算从 `ROW_ITEM_HEIGHT` 改为 `MAIN_TRACK_HEIGHT`
- 新建 `VideoTrackItem` 组件替代主轨中视频类型的 `TrackItem`

### 3. 新建 thumbnail-extractor.ts 模块

封装缩略图提取和管理逻辑：

- **computeThumbnailParams(pps, videoWidth, videoHeight, duration)**：
  - `thumbnailWidth = Math.round(40 * videoWidth / videoHeight)`
  - `step = thumbnailWidth / pps`
  - `count = Math.ceil(duration / step)`
  - 返回 `{ thumbnailWidth, step, count }`
- **class ThumbnailExtractor**：
  - `constructor(file: File)` — 创建 mediabunny Input 实例
  - `init()` — 异步初始化，获取 videoTrack、创建 CanvasSink（fit: "cover"），获取视频 width/height/duration（通过 mediabunny videoTrack 动态获取）
  - `extract(startTime, endTime, step)` — 生成 timestamps，调用 `sink.canvasesAtTimestamps()`，返回 `AsyncGenerator<{time, canvas}>`
  - `getVideoInfo()` — 返回 `{ width, height, duration }`
  - `dispose()` — 清理资源
  - 内部共享同一个 CanvasSink 对象，各行 slice 独立获取缩略图但共享 sink

### 4. 新建 VideoTrackItem.tsx 组件

命名为 `VideoTrackItem`（后续可能有更多自定义行为），负责单行中视频主轨的缩略图渲染：

- Props：`extractor: ThumbnailExtractor`、`visibleStart: number`、`visibleEnd: number`、`rowStartTime: number`、`pps: number`、`left: number`、`width: number`
- 使用 `<canvas>` 元素，宽度 = `(visibleEnd - visibleStart) * pps`，高度 = 40px
- 背景色 `#4A3728` 作为缩略图加载前的占位
- 通过 `createEffect` 监听 pps 变化（带 throttle 300ms），触发缩略图提取
- 提取完成后将缩略图绘制到 canvas：
  1. 根据 `computeThumbnailParams` 计算当前 pps 下的 step 和 thumbnailWidth
  2. 计算当前 slice 时间范围内的所有 timestamp
  3. 调用 `extractor.extract()` 获取缩略图
  4. 将每张缩略图 drawImage 到 canvas 对应位置

### 5. 集成到 TimelineRow.tsx

- 主轨渲染区域中，当 `item.type === "video"` 时，使用 `VideoTrackItem` 替代 `TrackItem`
- 传入当前行的 `visibleStart`、`visibleEnd`、`rowStartTime`、`pps`、`extractor` 等参数

### 6. ThumbnailExtractor 生命周期管理

- 在 `Timeline.tsx` 中，当 `mainTrackConf.item` 为视频类型时，创建 `ThumbnailExtractor` 实例
- 使用 `onCleanup` 在组件卸载时调用 `dispose()` 释放资源
- 通过 props 将 extractor 实例传递给 `TimelineRow` → `VideoTrackItem`
- 各行的 VideoTrackItem 共享同一个 extractor（共享 CanvasSink 对象）

### 7. Throttle 策略

- 当 `pps` 变化时，使用 throttle（300ms）延迟触发重新提取
- ThumbnailExtractor 内部维护已提取帧的缓存（Map<stepKey, Canvas>），pps 变化导致 step 变化时清除旧缓存
- 折行布局所有行都可见，无需额外的 IntersectionObserver 懒加载

### 8. 跨行渲染

视频 Item 通过 `assignItemsToRows()` 被拆分为多个 `RowItemSlice`，每个 slice 有 `visibleStart` 和 `visibleEnd`。每行的 `VideoTrackItem` 组件独立渲染各自时间段的缩略图，共享同一个 ThumbnailExtractor（CanvasSink 对象），天然支持跨行。

## Bug 分析

### 新增 bug

请重新审视缩略图刷新逻辑（@src/lib/VideoTrackItem.tsx），修复以下bug:

- 当前离开可视区域的 VideoTrackItem 会丢弃缩略图，滚动到可视区域内时重新加载
  - 期望初始加载可视区域内的缩略图，滚动时动态加载可视区域外的缩略图
  - 已加载的缩略图应该缓存，离开可视区域不能丢弃，只有失效（Pixels per second、容器高度变化）后才能丢弃
- `deleteRanges` 变化时，仍然触发缩略图刷新
  - 期望仅当Pixels per second、容器高度变化 时，触发缩略图刷新

### Bug1: 缩略图未懒加载

**现象**：所有行的 VideoTrackItem 组件一创建就立即触发缩略图提取，即使行在视口之外不可见。

**根因**：VideoTrackItem 的 `createEffect` 在组件挂载后立即执行（300ms throttle 后），没有可见性判断。

**修复方案**：在 VideoTrackItem 中使用 IntersectionObserver 监听根 div 元素是否进入视口，仅在可见时触发缩略图提取。离开视口时保留已渲染内容（无需清除）。

### Bug2: deleteRanges 变化触发缩略图刷新

**现象**：添加/删除区间时，缩略图会重新提取。

**根因**：`VideoTrackItem.tsx:72` 的 `createEffect(on(...))` 中 deps 使用了 `() => [props.pixelsPerSecond, props.visibleStart, props.visibleEnd]` 单函数返回数组的形式。SolidJS 的 `on()` 对单函数返回值使用 `===` 比较，而每次返回的数组都是新引用，导致任何上游响应式变化都会触发回调。

当 deleteRanges 变化 → `props.data` 新对象 → `mainSlicesByRow` memo 重算 → `slice()` 返回新对象 → props getter 重新求值 → deps 函数重新执行 → 返回新数组引用 → `on()` 判定为变化 → 触发 throttledDraw。

**修复方案**：将 `on()` 的 deps 从单函数返回数组改为数组形式 `[() => props.pixelsPerSecond, () => props.visibleStart, () => props.visibleEnd]`，使 SolidJS 逐元素比较基本类型值，避免不必要的重新触发。

### Bug3: specs 目录文件变更触发 HMR

**现象**：编辑 `specs/` 目录下的 md 文件导致前端页面重载。

**根因**：Vite 默认监听项目根目录下所有文件变更，未排除 specs 目录。

**修复方案**：在 `vite.config.ts` 中配置 `server.watch.ignored` 排除 `specs/` 目录。
