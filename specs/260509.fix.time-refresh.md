## 日志分析结论

### 关键发现

**1. `mainTrackConf.item` 引用未变，但 effect 仍然触发**

日志明确显示：

```
item引用是否变化: false
```

但 `[Timeline] mainTrackConf.item effect 触发` **仍然执行了**。

这说明 Solid 的 `on(() => props.initData.mainTrackConf.item, ...)` 在 **`props.initData` 整体引用变化**时，即使属性链上的值引用没变，effect 的 getter 也会被重新求值。Solid 的 `on` 默认使用 `===` 比较，理论上引用不变不应触发回调——但实际上**确实触发了**。

原因在于：`props` 在 Solid 中是响应式 proxy，当父组件 `setData` 创建新对象后，`props.initData` 变化导致所有依赖 `props.initData.xxx` 的 effect 的**依赖追踪**重新执行。虽然 getter 返回值相同（`===`），但 Solid 在某些场景下（如 `on` 配合 proxy props）无法跳过回调执行。

**2. 拖动时 extractor 每帧被销毁并重建**

移动素材日志清楚显示：

- 第1次 mousemove：`extractor: ThumbnailExtractor` → 正常
- 第2次 mousemove：`extractor: undefined` → 上一次 effect 执行中 `extractor()?.dispose()` + `setExtractor(undefined)` 已将提取器销毁
- 异步 `ext.init().then(() => setExtractor(ext))` 还没来得及完成，下一帧又触发了 effect，再次销毁

这导致了**永久性的 extractor 闪烁**：永远无法完成初始化，波形图和缩略图永远在重建。

**3. `<For>` 渲染函数被全部重新执行**

每次 items 变化，29 个 row 的渲染函数全部重新执行，每个 VideoTrackItem 的 effect 也全部触发。即使只有 1 个 item 被拖动，所有 29 个 row 都受到影响。

**4. 添加素材场景：缩略图确实被重新提取**

```
16x [VideoTrackItem] drawThumbnails 调用 hasDrawn: false lastDrawnPps: 0
```

所有 VideoTrackItem 的 `hasDrawn` 为 false（被重建后重置），触发了 16 次 `drawThumbnails`。

### 根因总结

| #   | 根因                                                                                                        | 日志证据                                                                    | 严重程度     |
| --- | ----------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- | ------------ |
| 1   | **`on(() => props.initData.mainTrackConf.item, ...)` 在 item 引用不变时仍然触发回调**                       | `item引用是否变化: false` 但 effect 仍执行，每次都 dispose + 重建 extractor | **严重**     |
| 2   | **`emitChange` → `App.setData` → `props.initData` 变化形成循环**：拖动时每帧触发                            | 每次 mousemove 都触发完整链路，extractor 反复销毁重建                       | **严重**     |
| 3   | **`<For>` 渲染函数全部重执行，VideoTrackItem 组件重建**                                                     | 29 row 全部重新渲染，`hasDrawn` 重置为 false                                | **中等**     |
| 4   | **`mainTrackConf` 引用未变**（`mainTrackConf引用是否变化: false`），但 Solid props proxy 的依赖追踪仍然触发 | onChange 回调中 mainTrackConf 引用没变                                      | **根本原因** |

## 修复方案

### 修复1：将 extractor 的 `createEffect` 改为仅依赖真正的业务数据（核心修复）

**问题**：`on(() => props.initData.mainTrackConf.item, ...)` 即使 item 引用不变也触发回调。

**方案**：用一个内部 signal 存储 mainTrackConf.item 的关键标识（如 `file` 引用），只在 file 真正变化时才重建 extractor。

```tsx
// 替代方案：使用 file 引用作为依赖
const mainTrackFile = createMemo(() => {
  const item = props.initData.mainTrackConf.item;
  return "file" in item ? item.file : null;
});

createEffect(
  on(mainTrackFile, (file) => {
    extractor()?.dispose();
    setExtractor(undefined);
    if (file) {
      const ext = new ThumbnailExtractor(file);
      ext.init().then(() => setExtractor(ext));
    }
  }),
);
```

对 waveformExt 同理。

### 修复2：拖动时不走外部 `onChange` → `setData` 循环

**问题**：拖动时每次 mousemove 调用 `emitChange` → `App.setData` → `props.initData` 变化 → 触发所有 effect。

**方案**：拖动过程中只更新内部 `setItems`，在 `mouseup` 时才调用 `emitChange` 通知外部。

```tsx
const onMove = (me: MouseEvent) => {
  // ...计算 newStart...
  const newItems = items().map(...);
  setItems(newItems);
  // 拖动中不调用 emitChange，避免触发外部 setData 循环
};

const onUp = () => {
  // 只在拖动结束时通知外部
  emitChange();
  // ...
};
```

### 修复3：`<For>` 内避免立即求值导致的子组件重建（可选优化）

**问题**：`mainSlice()` / `extractor()` 立即求值后传入子组件，每次 `<For>` 重渲染都产生新引用。

**方案**：将响应式数据通过 getter 函数传递，让子组件自行订阅：

```tsx
// TimelineRow 改为接收 getter
<TimelineRow
  getMainTrackSlice={() => mainSlicesByRow().get(row.rowIndex)?.[0]}
  getThumbnailExtractor={() => extractor()}
  ...
/>
```

或在 `<For>` 渲染函数内用 `createMemo` 缓存：

```tsx
<For each={rows()}>
  {(row) => {
    const slice = createMemo(() => mainSlicesByRow().get(row.rowIndex)?.[0]);
    const ext = createMemo(() => extractor());
    return <TimelineRow mainTrackSlice={slice()} thumbnailExtractor={ext()} ... />;
  }}
</For>
```

### 修复优先级

1. **修复1**（最高）：extractor 依赖改为 file 引用，避免不必要的销毁重建
2. **修复2**（高）：拖动中不调用 emitChange，消除循环
3. **修复3**（中）：`<For>` 内响应式优化，减少不必要的子组件更新

### 拖拽音频波形刷新根因（Bug 2）

拖拽覆盖层音频素材时，`item.startTime` 变化 → 波形图被不必要地重新渲染。

根因链路：

1. **`onMove` 调用 `setItems(newItems)`**（Timeline.tsx L246）
   - `items()` 变化 → `itemsByRow()` 重算 → `layersByRow()` 重算
   - `assignItemsToRows` 每次创建新的 `RowItemSlice` 对象

2. **TimelineRow 的 `<For each={effectiveLayers()}>` → `<For each={layer}>` 使用引用比较**
   - 新的 `RowItemSlice` 对象 ≠ 旧对象（`===` false）
   - `<For>` 视为新元素 → **回调重新执行 → AudioTrackItem 组件重建**
   - `hasDrawn` 重置为 false → `loadWaveform()` 触发 → 波形重绘

3. **`itemWaveformExtractors` effect 在每次 `items()` 变化时触发**（Timeline.tsx L166-199）
   - 即使音频 item 集合没变（仅位置变化），仍创建新 Map 并 `setItemWaveformExtractors(next)`
   - 导致 `itemWfExts` memo 传播变化 → TimelineRow 收到新 `itemWaveformExtractors` prop

### 核心问题

Solid 的 `<For>` 按引用追踪元素。`RowItemSlice` 对象在每次 layout 计算中被重新创建，导致 `<For>` 认为所有元素都是新的，重建组件。使用 `<Index>` 按位置追踪可避免组件重建，仅更新 props。

## 技术实现方案

### 方案1：overlay `<For>` 改为 `<Index>`（核心修复）

将 TimelineRow.tsx 中覆盖层渲染的嵌套 `<For>` 改为 `<Index>`：

- `<Index>` 按数组位置追踪元素，回调不会因元素引用变化而重建
- 元素内容变化时，回调中的 accessor 更新 → props 响应式更新 → 组件不重建
- AudioTrackItem 的 `hasDrawn` 状态保留 → 波形不重绘

关键变化：
- `slice` 从直接值变为 accessor `slice()`
- `layerIdx` 从 accessor `layerIdx()` 变为 plain number `layerIdx`

### 方案2：`itemWaveformExtractors` effect 优化

在 `setItemWaveformExtractors(next)` 前检查 Map 内容是否相同：
- 如果 key 集合和 value 引用都相同，跳过更新
- 拖拽时 audio item 集合不变 → effect 运行但不触发 signal 更新 → 下游不受影响
