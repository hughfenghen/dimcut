## 现状分析

### Bug 1：播放状态卡死

**根因**：`isPlaying` 状态（由 `App.tsx` 持有）与 `<video>` 元素实际播放状态脱节。

完整链路：

1. 用户点击播放 → `isPlaying` 设为 `true` → `PreviewPlayer` 内 effect 调用 `el.play()` + `startRAF()`
2. 用户从时间轴删除片段 → `App.tsx` 调用 `setData` 更新 items/deletedRanges
3. `PreviewPlayer` 收到新的 `deletedRanges`，在 `skipIfDeleted` 中跳过已删除区间（设置 `el.currentTime = r.end`）
4. 若跳转导致视频暂停（浏览器在 `currentTime` 赋值时可能触发 pause），或 `r.end` 到达末尾触发 `ended` 事件，`<video>` 实际暂停
5. 此时 `isPlaying` 仍为 `true`（没有代码在 `video.pause` 事件时同步回父组件）
6. 用户再次点击播放按钮 → `togglePlay` 检查 `el.paused === true` → 调用 `onPlayPause(true)`
7. 但 `isPlaying()` 已经是 `true`，信号值未变化，SolidJS 的 `createEffect(on(() => props.isPlaying, ...))` 不触发 → 视频不恢复播放

**关键代码位置**：

- `PreviewPlayer.tsx:171-179` — `togglePlay` 基于 `el.paused` 判断，而非 `props.isPlaying`
- `PreviewPlayer.tsx:127-143` — `isPlaying` effect 不会在信号值相同时重新触发
- `PreviewPlayer.tsx:145-161` — `startRAF` 中 `el.currentTime` 赋值可能导致浏览器暂停

### Bug 3：删除片段时页面闪烁

**根因**：与 Bug 4 同源 —— `App.tsx` 的 timeline `onChange` 在每次变更时都会调用 `setData((d) => ({ ...d, items: arg.items! }))`，导致 `data` 信号触发新引用；`PreviewPlayer.tsx` 的 `createEffect(on(() => props.mainTrackConf.item.file, ...))` 中 SolidJS 的 `on` 默认只追踪依赖信号触发、并不对返回值做相等比较，因此即便 `file` 引用未变，effect 体也会重新执行：撤销旧 ObjectURL → 创建新 ObjectURL → `<video>` 的 `src` 被替换 → 浏览器把元素的整段帧画面清空并重新加载首帧，呈现为一次黑屏闪烁。

用户在 bug 描述中的猜测（"deletedRanges 更新组件重新渲染、重新布局导致闪烁"）已排除：

- `rows()` memo 仅依赖 `totalDuration / containerWidth / pps`，不依赖 `deletedRanges` 或 `items`，因此 timeline 的整体行布局不会因删除片段而重排
- `DeletedRangeOverlay` 仅是绝对定位的覆盖层，不挤占其他元素

闪烁的实际来源是右侧预览视频的整段 reload，而不是左侧 timeline 的重排。

**关键代码位置**：

- `App.tsx:402-407` — `onChange` 无条件以新引用更新 `data`
- `PreviewPlayer.tsx:40-49` — `on(() => props.mainTrackConf.item.file, ...)` 在 `data` 触发时也会重跑，重建 ObjectURL

### Bug 4：删除片段后预览重置到 0s 暂停

**根因**：与 Bug 3 完全同源。`<video>.src` 被替换为新的 ObjectURL 时，浏览器会：

1. 把 `currentTime` 复位为 0，并通过 `loadstart` / `emptied` 状态触发 `pause` 事件
2. `pause` 事件命中 Bug 1 修复中新增的 `handlePause`，因 `props.isPlaying === true` 而调用 `props.onPlayPause(false)` 把状态同步回父组件
3. RAF 在 stopRAF 之前可能再跑一次 tick，将 `el.currentTime`（已经是 0）通过 `props.onTimeUpdate(0)` 同步给 `App.tsx`，使 `currentTime` 信号被改写为 0
4. 用户再次点击播放：`isPlaying` 变为 `true` → `PreviewPlayer` 的 `isPlaying` effect 执行 `el.play()`，此时 `el.currentTime` 已是 0，所以从 0s 开始播

因此本质上不是 "保留位置" 的 bug，而是 "本不该发生的 reload 把位置擦掉了" 的 bug。修复 Bug 3 的 reload 问题后，删除片段不再 reload 视频，currentTime 与 isPlaying 都会自然保持。

**关键代码位置**：

- `PreviewPlayer.tsx:40-49` — 重建 ObjectURL 导致视频 reload
- `PreviewPlayer.tsx:145-162` — `startRAF` 的 tick 在 reload 期间可能把 0 写回 currentTime
- `PreviewPlayer.tsx:171-176` — `handlePause` 把 reload 引发的 pause 当成意外暂停同步回父组件

### Bug 2：落地页 sticky 无效

**根因**：`DemoSection.tsx` 的 `<section>` 元素设置了 `overflow-hidden`，CSS `position: sticky` 在任何祖先设置了非 `visible` 的 `overflow` 时失效。

DOM 层级：

```
<section class="overflow-hidden">     ← DemoSection.tsx:11，overflow-hidden 打断 sticky
  <div class="w-fit min-h-[...]">     ← 白色卡片容器
    <App>
      <div class="flex items-start">  ← flex 布局
        <div class="sticky top-4">    ← App.tsx:416，sticky 目标元素
```

**关键代码位置**：

- `DemoSection.tsx:11` — `overflow-hidden` 导致子元素 sticky 失效

## 技术实现方案

### Bug 1 修复

**方案**：在 `PreviewPlayer` 内监听 `<video>`/`<audio>` 元素的 `pause` 事件，当媒体元素非主动暂停时，将状态同步回父组件。

具体改动：

1. **`PreviewPlayer.tsx`**：添加 `onPause` 事件处理器到 `<video>` 和 `<audio>` 元素，当元素被暂停且当前 `props.isPlaying === true` 时，调用 `props.onPlayPause(false)` 同步状态
2. **`PreviewPlayer.tsx`**：修改 `togglePlay` 逻辑，基于 `props.isPlaying` 而非 `el.paused` 判断切换方向，确保即使状态不一致也能正确切换

### Bug 2 修复

**方案**：将 `overflow-hidden` 改为 `overflow-x-clip`（Tailwind v4.2.4，`overflow-x: clip` 不创建滚动容器，不影响 sticky）。

具体改动：

1. **`DemoSection.tsx`**：将 `<section>` 的 `overflow-hidden` 替换为 `overflow-x-clip`

### Bug 3 & Bug 4 修复（共用方案）

**方案**：在 `PreviewPlayer.tsx` 的 file 监听 effect 内增加引用相等比较，仅在 `file` 真正变化时才撤销旧 URL、重建新 ObjectURL；当 `App.tsx` 因 items/deletedRanges 变化触发 `data` 信号、但底层 `file` 引用未变时，跳过 reload。

为什么不在 `App.tsx` 侧避免新建 `data` 引用：

- `data` 是包含 `items` 的复合信号，items 变化时本来就必须用新引用，否则下游订阅不会感知到（SolidJS 用 `Object.is` 判定信号是否变更）
- 真正脆弱的点是 PreviewPlayer 把"父引用变化"当成了"file 变化"，应在该处加防御性比较，而不是反过来要求所有调用方都不许更新 data

具体改动：

1. **`PreviewPlayer.tsx`**：把 `createEffect(on(() => props.mainTrackConf.item.file, (file) => {...}))` 中的回调改为 `(file, prev) => { if (file === prev) return; ... }`。首次执行时 `prev` 为 `undefined`，仍会创建 ObjectURL；后续仅在 file 引用实际更换（如 `handleReplaceMainTrack`）时才执行 reload 逻辑。
2. 验证副作用：
   - `audioItems` effect 自身有 `currentIds` 增删判断，重复触发不会副作用泄漏
   - `currentTime` effect 在 `isSelfUpdate` 期间已自我屏蔽，与本修复无交互
   - Bug 1 修复中新加的 `handlePause` 不需要改动 —— 修复 reload 后视频不再因 reload 暂停，handlePause 自然不会被错误触发
