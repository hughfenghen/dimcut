## 现状分析

### 整体架构

波形图涉及三个核心模块：

1. **WaveformExtractor** (`src/lib/waveform-extractor.ts`)：音频解码 + 峰值计算
2. **AudioTrackItem / VideoTrackItem**：调用 extractor 获取波形数据，带 throttle 和 IntersectionObserver 懒加载
3. **WaveformCanvas** (`src/lib/WaveformCanvas.tsx`)：Canvas 绘制波形条

### 关键执行流程

1. `Timeline` 组件创建 `WaveformExtractor`，调用 `init()` —— 创建 `Input`、获取音频轨道、创建 `AudioBufferSink`
2. TrackItem 可见时调用 `extractor.extract(startTime, endTime, pps)`
3. `extract()` 检查缓存（按 pps 键值），未命中则调用 `buildFullPeaks(pps)` 解码**整段音频**
4. `buildFullPeaks()` 通过 `this.sink.buffers()` 异步迭代获取所有 PCM buffer，逐样本计算峰值

### 性能瓶颈分析

**瓶颈 1：每次都完整解码整段音频（核心瓶颈）**

- `buildFullPeaks()` 遍历 `this.sink.buffers()` 获取所有 PCM 数据
- 5 分钟音频 @ 44.1kHz 单声道 = ~1300 万样本，需要全部解码
- 与缩略图对比：缩略图只 seek 到目标时间戳抽帧（`canvasesAtTimestamps`），**不需要解码整段视频**；而波形图需要逐样本遍历
- 这是波形图比缩略图慢得多的根本原因

**瓶颈 2：pps 变化时重复解码（已确认 `buffers()` 支持多次调用）**

- `AudioBufferSink.buffers()` 支持多次调用，不存在迭代器耗尽问题
- 但 pps 变化时仍会 `cache.clear()` 并重新全量解码，应通过原始峰值缓存避免

**瓶颈 3：逐样本 pixelIndex 映射计算**

- `buildFullPeaks()` 中对每个采样点都计算 `sampleTime = bufferStartTime + i / sampleRate` 和 `pixelIndex = Math.floor(sampleTime * pps)`
- 这涉及大量浮点运算（1300 万次），虽然单次计算快，但累积耗时不可忽略
- 可以用整数批量处理优化

**瓶颈 4：VideoTrackItem 串行加载波形**

- `VideoTrackItem.drawThumbnails()` 在缩略图全部加载完后，才 `await` 波形提取
- 波形和缩略图是串行的，增加了整体加载时间

### 缓存策略评估

- 当前按 `pps` 键缓存完整波形数据，pps 变化时清除旧缓存仅保留新值
- 缓存设计合理（不同缩放级别需要不同精度的峰值），但问题是**首次构建代价太高**
- `extract()` 返回的是 `fullPeaks.slice()`，即裁切后的子视图，这部分没有性能问题

### 懒加载策略评估

- IntersectionObserver + 300ms throttle 已实现基本的懒加载
- 但 `buildFullPeaks()` 是全量解码，即使只显示一小段波形也需要解码整段音频
- 懒加载只延迟了触发时机，没有减少工作量

## 技术实现方案

### 方案概述

核心思路：**将全量 PCM 解码改为一次性解码并缓存原始峰值数据，后续不同 pps 通过降采样原始峰值得到，避免重复解码**。

### 方案详情

#### 1. 一次性解码 + 原始峰值缓存

将当前"按 pps 解码"改为"解码一次，缓存原始峰值"：

- `init()` 阶段解码整段音频，以固定分辨率（如每秒 1000 个峰值）计算原始峰值数组 `rawPeaks`
- 后续 `extract()` 根据目标 pps，从 `rawPeaks` 降采样得到结果，无需再次解码
- 优势：解码只发生一次，pps 变化时无需重新解码，降采样操作是纯数学计算，非常快

#### 2. 优化逐样本映射计算

在 `buildFullPeaks()` 中：

- 用 `samplesPerPixel = sampleRate / pps` 预计算每个 pixel 对应的样本数
- 按 pixel 批量处理而非逐样本计算 pixelIndex，减少浮点运算

#### 3. VideoTrackItem 并行加载波形与缩略图

- 将 `drawThumbnails()` 中波形加载从串行 `await` 改为与缩略图并行
- 波形加载独立触发，不等待缩略图完成

#### 4. 添加性能日志

- 在关键环节（init、解码、buildPeaks、extract slice）添加 `console.time` / `console.timeEnd`
- 用于验证优化效果

### Bug 分析：首屏波形图不可见

#### 现象

性能确实有较大提升（decode + buildRawPeaks: ~1.7s），但初始化完成后首屏内波形图不可见，滚动后第二屏或 PPS 变更后才正常。

#### 根因

`VideoTrackItem` 的 `createEffect` 只监听 `[pixelsPerSecond, isVisible]`，不监听 `waveformExtractor` prop：

1. 首屏 VideoTrackItem 可见 → effect 触发 → `drawThumbnails()`
2. 此时 `waveformExtractor` 仍为 `undefined`（init 含 ~1.7s buildRawPeaks 未完成），波形被跳过
3. `hasDrawn = true`, `lastDrawnPps = pps`
4. `waveformExtractor` prop 后续更新（init 完成），但 effect 不追踪 → 不再重绘
5. 直到 pps 变化或滚出再滚入才触发

#### 修复方案

在 `VideoTrackItem` 中添加独立 effect 监听 `waveformExtractor`：当其从 undefined 变为已初始化时，同步加载波形数据（无需重绘缩略图）
